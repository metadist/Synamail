### Step 3.1 pre-flight audit — BLOCKING REWRITE NEEDED

Audited `RealSynaplanClient` (`src/shared/synaplan-client.ts`) against the **live** Synaplan OpenAPI spec, checkpointed at `docs/openapi-2026-05-21.json` (OpenAPI 3.0, 285 paths, 777 KB). Both the client **and** `docs/FEATURES.md §6` contain assumptions that **don't match the real API**:

| Endpoint / behavior | Client assumes | Real spec | Severity |
| --- | --- | --- | --- |
| `POST /api/v1/messages/send` request body | `{action: 'summarise'\|'translate'\|'reply'\|'classify', input: {...}}` | `{message: string, trackId?: int}` — single free-form message field | 🚨 HIGH — all 4 read-mode features wired wrong |
| `POST /api/v1/messages/send` response | `SummariseResult` / `TranslateResult` / `DraftReplyResult` / `ClassifyResult` (typed per action) | `{success, incomingMessage: object, outgoingMessage: object}` — `outgoingMessage` carries the AI text reply | 🚨 HIGH — response parsing must extract AI text; for `classify`, JSON.parse the AI reply client-side |
| `POST /api/v1/chats` request body | `{conversationId, question, emailContext}` ("ask a question") | `{title?: string}` — just creates an empty chat | 🚨 HIGH — `ask()` needs split into create-chat + send-message-with-`trackId` |
| Chat follow-ups | `POST /api/v1/chats` per turn | Create chat once → send each turn via `POST /messages/send` with `trackId = chat.id`; read history via `GET /chats/{id}/messages` | 🚨 HIGH — affects `ask()` and any future chat-thread feature |
| `POST /api/v1/rag/search` body | `{query, threshold, limit, groups: string[]}` | `{query, limit, min_score, group_key: string\|null}` — **singular** group, snake_case, `min_score` not `threshold` | 🚨 HIGH — multi-group search not supported; breaks the `contact:<email>` multi-scope assumption |
| `POST /api/v1/files/upload` | JSON `{filename, contentBase64, mimeType, groupId, metadata}` | `multipart/form-data` with `files[]`, `group_key`, `process_level` (enum: `store`\|`extract`\|`vectorize`\|`full`) | 🚨 HIGH — switch to FormData + base64 → Blob decode |
| `POST /api/v1/files/{id}/process` separate call | Required after upload | **Not required** — upload accepts `process_level` directly, combining the two steps | 🟡 MEDIUM — simplification opportunity |
| `POST /api/v1/files/groups` (create group) | Exists | **Does not exist** — `/files/groups` is GET-only | 🚨 HIGH — groups are created implicitly when an upload uses a new `group_key`. `ragCreateGroup()` becomes a no-op + roaming-stored pending name, or moves out of the client interface entirely. "Create new group" UX needs reframing as "type a name → next upload uses it" |
| `GET /api/v1/profile` response | `{ok, email}` | `{success, profile: {email, firstName, lastName, phone, …}}` | 🟡 MEDIUM — `ping()` is currently broken; would crash on `.email` |
| Alternative ping target | n/a | `GET /api/v1/auth/me` → `{success, user: {id, email, level, isAdmin, …}}` — purpose-built for "who am I right now" | 🟢 LOW — switch to `/auth/me` for ping |
| `DELETE /api/v1/apikeys/{keyId}` path param | `{keyId}` | `{id}` | 🟢 LOW — cosmetic, positional |

#### Recommended Step 3.1 plan (revised scope)

Original estimate was 0.5 day for "swap mock for real". The audit reveals it's a **2–3 day reshape**:

1. **Rewrite `synaplan-client.ts`** to match the real contract:
   - `summarise`/`translate`/`draftReply`/`classify` → `{message: systemPrompt + '\n\n' + emailBody}` to `/messages/send`; extract AI text from `outgoingMessage`; for `classify`, `JSON.parse` the reply with a Zod fallback.
   - `ask` → ensure-chat-exists per `conversationId` (cache `chatId` in roaming `chats` map already on the type), then `/messages/send` with `trackId`.
   - `ragSearch` → single `group_key`; if multi-group really needed, loop client-side or accept the v1 limitation.
   - `fileUpload` → `FormData` (`files[]`, `group_key`, `process_level`); decode base64 → `Blob` client-side. Drop the separate `fileProcess` step (or keep for `re-vectorize` only).
   - `ragCreateGroup` → no-op + roaming pending name, or remove from interface.
   - `ping` → read `profile.email` (or switch to `/auth/me`).
   - `revokeApiKey` → path param `{id}` (cosmetic, but tidy up).
2. **Re-shape `src/shared/types.ts`** to match (most result types collapse to "AI text reply" + classify-specific JSON shape).
3. **Update `docs/FEATURES.md §6`** with corrected endpoint mapping.
4. **One live smoke round-trip** of each endpoint with a real API key — necessary because `outgoingMessage` is `type: object` (undocumented internal shape) in the OpenAPI; we need a real response to know exactly which field carries the AI text.
5. **THEN** decide codegen strategy (`openapi-zod-client` / `openapi-zodios` / hand-written). At that point we'll know which schemas to generate.
6. **Vitest**: header injection, retry-on-5xx, 401 → clear-and-bounce, classify JSON-parse failure, response shape variants.
