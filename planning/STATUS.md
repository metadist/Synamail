### 🔑 Important findings of today (read before debugging again)

1. **Synaplan local vectorization was silently failing on a Cloudflare 401.** The user's
   `VECTORIZE` model default (`BCONFIG DEFAULTMODEL`, **owner-1 row overrides the owner-0
   global**) pointed at a Cloudflare embedding model with an invalid token → every
   embed/RAG call 401'd → Qdrant `user_documents` collection never got created
   ("collection doesn't exist") → nothing in the knowledge base. **Fix:** point VECTORIZE
   at the local Ollama `bge-m3` (BID 13). **Lesson:** check **per-user** model overrides,
   not just the global defaults.
2. **Synaplan file ingestion rejects `.eml` / `message/rfc822`.** Allowed types:
   `pdf, docx, doc, xlsx, xls, pptx, ppt, txt, md, csv, odt, …, jpg, png, …, mp3, mp4, …`.
   Synamail was exporting read emails as `.eml` → rejected. **Fix:** upload **plain text**
   (subject/headers + body) — also better for RAG than a MIME envelope.
3. **A "success" toast lied.** Synamail's `fileUpload` ignored `success:false` + the
   `errors` array (HTTP 206) and reported **`file #0`**. Always **verify with the backend
   (DB + logs + a curl repro)** when something "succeeds" but doesn't appear.
4. **EWS XML local-name collision:** `<s:Body>` (SOAP envelope) and `<t:Body>` (message
   body) share local name `Body`. A document-wide `getElementsByTagName`-style search
   grabs the SOAP body. **Scope parsing to the `<t:Message>` element.**
5. **Dark-mode color tokens must be defined for BOTH schemes.** `--syn-success`/`-danger`
   had no dark override → dark-green text on dark-grey surface. Never put semantic text
   on `--syn-surface` without a paired `-bg`/`-fg` token override.
6. **Office taskpane is not a background service** — Mail Routes fire add-in-driven on
   `ItemChanged`, never silently in the background (already locked in `MAIL_ROUTES.md`).

### ⚠️ UI persistency problem (flagged today — tackle tomorrow)

**Problem.** Office taskpanes **reload on every open**, and Synamail's UI state is
in-memory only. So reopening the pane loses: accordion **open/closed** state
(`AccordionItem :open` only seeds the _initial_ state — user toggles aren't remembered),
the **Home chat thread** (only `chatId` is persisted in roaming; the messages array is
in-memory), ReadMode/EmailActions **ask history**, the **Compose answer** prompt, and the
**Knowledge filter** query/results. Anything typed mid-task vanishes on reopen.

**Suggestions for tomorrow.**

1. Add a small `useUiState` composable backed by `roamingSettings` (with the existing
   `localStorage` fallback), namespaced `synamail.ui.*`. Keep writes debounced.
2. Persist per-flap open/closed (`synamail.ui.flaps`) — but honour "all closed on open"
   as the default unless the user pinned one.
3. Persist in-progress inputs (compose-answer prompt, ask question, KB filter query +
   selected group) and restore on mount.
4. Rehydrate the Home chat: persist the last N messages for the `home` conversation, OR
   refetch via `GET /chats/{id}/messages` using the stored `chatId`, so the thread
   survives a reopen.
5. **Test determinism:** persistence must work offline via the localStorage fallback and
   must not depend on `.env`/locale/timezone (per AGENTS.md).

#### Recommended Step 3.1 plan (revised scope)

Original estimate was 0.5 day for "swap mock for real". The audit reveals it's a **2–3 day reshape**:

1. **Rewrite `synaplan-client.ts`** to match the real contract:
   - `summarise`/`translate`/`draftReply`/`classify` → `{message: systemPrompt + '\n\n' + emailBody}` to `/messages/send`; extract AI text from `outgoingMessage`; for `classify`, `JSON.parse` the reply with a Zod fallback.
   - `ask` → ensure-chat-exists per `conversationId` (cache `chatId` in roaming `chats` map already on the type), then `/messages/send` with `trackId`.
   - `ragSearch` → single `group_key`; if multi-group really needed, loop client-side or accept the v1 limitation.
   - `fileUpload` → `FormData` (`files[]`, `group_key`, `process_level`); decode base64 → `Blob` client-side. Drop the separate `fileProcess` step (or keep for `re-vectorize` only).
   - `ragCreateGroup` → no-op + roaming pending name, or remove from interface.
   - `ping` → read `profile.email` (or switch to `/auth/me`).
   - `revokeApiKey` → path param `{id}` (cosmetic, but tidy up).
2. **Re-shape `src/shared/types.ts`** to match (most result types collapse to "AI text reply" + classify-specific JSON shape).
3. **Update `docs/FEATURES.md §6`** with corrected endpoint mapping.
4. **One live smoke round-trip** of each endpoint with a real API key — necessary because `outgoingMessage` is `type: object` (undocumented internal shape) in the OpenAPI; we need a real response to know exactly which field carries the AI text.
5. **THEN** decide codegen strategy (`openapi-zod-client` / `openapi-zodios` / hand-written). At that point we'll know which schemas to generate.
6. **Vitest**: header injection, retry-on-5xx, 401 → clear-and-bounce, classify JSON-parse failure, response shape variants.
