#### Recommended Step 3.1 plan (revised scope)

Original estimate was 0.5 day for "swap mock for real". The audit reveals it's a **2вЂ“3 day reshape**:

1. **Rewrite `synaplan-client.ts`** to match the real contract:
   - `summarise`/`translate`/`draftReply`/`classify` в†’ `{message: systemPrompt + '\n\n' + emailBody}` to `/messages/send`; extract AI text from `outgoingMessage`; for `classify`, `JSON.parse` the reply with a Zod fallback.
   - `ask` в†’ ensure-chat-exists per `conversationId` (cache `chatId` in roaming `chats` map already on the type), then `/messages/send` with `trackId`.
   - `ragSearch` в†’ single `group_key`; if multi-group really needed, loop client-side or accept the v1 limitation.
   - `fileUpload` в†’ `FormData` (`files[]`, `group_key`, `process_level`); decode base64 в†’ `Blob` client-side. Drop the separate `fileProcess` step (or keep for `re-vectorize` only).
   - `ragCreateGroup` в†’ no-op + roaming pending name, or remove from interface.
   - `ping` в†’ read `profile.email` (or switch to `/auth/me`).
   - `revokeApiKey` в†’ path param `{id}` (cosmetic, but tidy up).
2. **Re-shape `src/shared/types.ts`** to match (most result types collapse to "AI text reply" + classify-specific JSON shape).
3. **Update `docs/FEATURES.md В§6`** with corrected endpoint mapping.
4. **One live smoke round-trip** of each endpoint with a real API key вЂ” necessary because `outgoingMessage` is `type: object` (undocumented internal shape) in the OpenAPI; we need a real response to know exactly which field carries the AI text.
5. **THEN** decide codegen strategy (`openapi-zod-client` / `openapi-zodios` / hand-written). At that point we'll know which schemas to generate.
6. **Vitest**: header injection, retry-on-5xx, 401 в†’ clear-and-bounce, classify JSON-parse failure, response shape variants.
