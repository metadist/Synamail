# Commit Process

The professional commit, review, and release process for Synamail. Mirrors the gate structure used in `@/wwwroot/synaplan`, adapted for a TypeScript-only Vue/Office.js codebase.

> **Single rule that overrides everything else:** _never_ commit code that hasn't passed the pre-commit gate (see В§3). Everything in this file exists to make that rule pleasant to follow.

## 1. Conventional Commits

Every commit message **must** follow [Conventional Commits 1.0](https://www.conventionalcommits.org/en/v1.0.0/):

```
<type>(<scope>)?: <imperative-mood subject>

<optional body>

<optional footer>
```

### Allowed types

| Type       | Use for                                                                     |
| ---------- | --------------------------------------------------------------------------- |
| `feat`     | A new user-visible feature.                                                 |
| `fix`      | A bug fix.                                                                  |
| `refactor` | Code change that neither fixes a bug nor adds a feature.                    |
| `perf`     | A change that improves performance.                                         |
| `test`     | Adding or correcting tests; no production-code change.                      |
| `docs`     | Documentation-only changes (`README.md`, `docs/`, `planning/`, JSDoc).      |
| `build`    | Build-system / dependency / Vite config changes.                            |
| `ci`       | `.github/workflows/*`, Makefile changes, hook scripts.                      |
| `chore`    | Housekeeping with no source-code impact (licence header, formatting sweep). |
| `style`    | Code formatting only (whitespace, semicolons) вҖ” usually CI-driven.          |
| `revert`   | Revert of a previous commit (auto-generated by `git revert`).               |

### Scope (optional but encouraged)

Pick from this fixed list to keep history searchable:

`taskpane`, `commands`, `dialog`, `auth`, `client`, `manifest`, `i18n`, `gui`, `rag`, `rules`, `contact-profile`, `tests`, `ci`, `docs`, `deps`, `assets`.

### Subject rules

- **Imperative mood:** "add login button", not "added" or "adds".
- **No trailing period.**
- **вүӨ 72 chars.**
- **English.**

### Examples

```
feat(taskpane): add Summarise button to ReadMode

fix(auth): clear roaming settings on persistent 401

docs(plan): split PLAN into 4 sprints

ci: validate manifest.xml on every PR

refactor(client)!: switch from axios to fetch
```

A `!` after the type (`refactor(client)!:`) or a `BREAKING CHANGE:` footer marks a breaking change. Both are valid.

### Body

- Wrap at 80 chars.
- Explain **the why**, not the what (the diff already shows the what).
- Reference issues with `Refs #123`, close with `Fixes #123`.

### Footers

- `Refs:`, `Fixes:`, `Closes:` вҖ” issue links.
- `BREAKING CHANGE:` вҖ” describe the break and the migration.
- **Never:** "Generated with вҖҰ", "Co-Authored-By: <AI tool>", "Signed-off-by: <AI tool>", or any AI-attribution footer.

## 2. Branches & PRs

### Branch naming

```
<type>/<short-kebab-summary>
```

Example: `feat/contact-profile`, `fix/auth-401-loop`, `docs/sprint-3-spec`.

### Branch policy

| Branch               | Direct push? | Force push?                  | Source for                          |
| -------------------- | ------------ | ---------------------------- | ----------------------------------- |
| `main`               | вқҢ           | вқҢ                           | All releases                        |
| `develop` (optional) | вқҢ           | вқҢ                           | Integration of feature work         |
| `feat/*`, `fix/*`, вҖҰ | вң… (author)  | вң… (author only, pre-review) | Open PR back to `develop` or `main` |

Branch protection on `main` requires:

- Green CI (gate job `all-checks-passed`).
- 1 review from a CODEOWNERS approver.
- Up-to-date with target branch.
- No outstanding "request changes" reviews.

### Pull-request rules

- One logical change per PR. If you can't write a one-sentence summary, split it.
- PR title **also** follows Conventional Commits.
- Use the `PULL_REQUEST_TEMPLATE.md` вҖ” fill every section honestly. "Not tested" is acceptable with a reason; silence is not.
- Link the issue with `Fixes #N` so it auto-closes on merge.
- After review feedback: push **new commits**, not force-pushes (reviewers need to see what changed).
- Once approved and CI green: **Squash-merge** with the PR title as the merge commit message. The body of the squash commit is the PR description.

### Reviewer rules

- Reviewer leaves comments inline.
- "Approve" only if you'd ship the diff yourself.
- "Request changes" if a comment is blocking; otherwise "Comment".
- Self-review your own PR before requesting reviewers.

## 3. Pre-commit gate

Run **before every `git commit`** and **before every `git push`**. The hook in `.githooks/pre-commit` runs `make ci-local` for you; do not bypass it with `git commit --no-verify` unless you have a tracked issue ID.

```bash
make ci-local
```

That target chains:

1. `make lint-docs` вҖ” markdownlint, the same command CI's `Docs lint` job runs.
2. `make lint` вҖ” Prettier + ESLint.
3. `make check-types` вҖ” `vue-tsc -b`.
4. `make test` вҖ” Vitest unit + component.
5. `make validate` вҖ” `office-addin-manifest validate manifest.xml`.
6. `make build` вҖ” Vite production build, fails on bundle-size budget.

For E2E (Playwright) вҖ” not part of the local pre-commit gate; runs in CI on PR. But run it manually when you touch:

- Auth flow (`useAuth.ts`, `dialog/`).
- Synaplan client (`synaplan-client.ts`).
- Any view.

```bash
make test-e2e
```

- `lint` is the first gate вҖ” fast, catches the most common issues.
- `types` + `manifest` run in parallel after lint.

- `build` produces the `dist/` artefact uploaded for sprint reviews and AppSource staging.
- `e2e` runs only on pull requests (and on tag pushes for releases). Tagged with `@ci` for the on-PR subset; full suite runs nightly.
- `all-checks-passed` is the single required check for branch protection вҖ” if you add a stage, add it to this gate's `needs:` list.

### Dependabot / Renovate

- Dependabot (`.github/dependabot.yml`) keeps `npm`, GitHub Actions, and Docker bases up to date.
- Group minor + patch updates into a single weekly PR per ecosystem.
- Major updates open individually so they get human review.

## 5. Releases

Versioning: **[SemVer 2.0](https://semver.org/)** вҖ” `MAJOR.MINOR.PATCH`. Four-part `MAJOR.MINOR.PATCH.BUILD` for the manifest version (Microsoft requires four parts).

### Release flow

1. All work for the release is merged into `main`, CI green.
2. Cut a release branch: `release/v1.2.0`.
3. Bump `package.json` `version` and `manifest.xml` `<Version>` (four parts: `1.2.0.0`).
4. Run `make ci-local` and `make test-e2e` locally on the release branch.
5. Tag the commit: `git tag -a v1.2.0 -m "Release v1.2.0"` and push.
6. CI builds the production manifest + dist artefact and uploads to the GitHub Release.
7. AppSource submission (Sprint 4 onwards) pulls the tagged artefact.

### Hotfixes

1. Branch from the affected tag: `git checkout -b hotfix/v1.2.1 v1.2.0`.
2. Fix, run the gate, PR to `main`.
3. After merge, cherry-pick into `develop` (if used).
4. Tag `v1.2.1` and re-submit if the production manifest needs an update.

## 6. Local hooks

Hooks live in `.githooks/` and are enabled by:

```bash
git config core.hooksPath .githooks
```

This is **a manual one-time setup** documented in `README.md`. The `make bootstrap` target runs it automatically once `package.json` exists (Sprint 2.1).

Active hooks:

| Hook         | Action                                                                                                                                                                           |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `pre-commit` | Runs `make ci-local` (or `make lint-docs` вҖ” the exact markdownlint CI runs вҖ” for docs-only changes, auto-detected).                                                              |
| `commit-msg` | Validates the commit message against Conventional Commits (regex shipped in the hook). Once `npm` is available, swap for `commitlint` (`commitlint.config.cjs` already present). |
| `pre-push`   | Reminds you to run `make test-e2e` if any of `useAuth.ts`, `synaplan-client.ts`, or `src/taskpane/views/` changed in your branch.                                                |

### Bypassing the hook

Allowed in **exactly one case**: you are recording a commit that documents a known broken state (e.g. mid-bisect, mid-merge-conflict reproduction). Prefix the commit subject with `wip:` and **never push that commit**. Squash it out before opening a PR.

## 7. AI-agent rules

Recap of the rules in `AGENTS.md` for completeness, with this file as the canonical source for the commit process:

- AI agents follow this entire document literally.
- AI agents never bypass `pre-commit` or `commit-msg`.
- AI agents never add attribution footers ("Generated with вҖҰ", "Co-Authored-By: <AI tool>").
- AI agents never force-push.
- AI agents never `git checkout --ours` / `--theirs` to resolve a merge conflict.
- AI agents never commit `.env`, secrets, or signed production manifests.

## 8. Cheat sheet

```bash
# Start a feature
git checkout -b feat/contact-profile-search

# Work, test, commit
make ci-local
git add -A
git commit -m "feat(contact-profile): add sender-scoped RAG search"

# Push and open a PR
git push -u origin feat/contact-profile-search
gh pr create --fill

# Update against main
git fetch origin
git rebase origin/main          # preferred
make ci-local                   # re-run the gate

# Merge (squash) once reviewed + green
gh pr merge --squash --delete-branch
```
